# Runbook — Bootstrap (zero → running platform)

Audience: platform operators standing up an environment from nothing. Companion
runbooks: [UPGRADE.md](UPGRADE.md), [RECOVERY.md](RECOVERY.md),
[TEARDOWN.md](TEARDOWN.md). Design context: [ARCHITECTURE.md](ARCHITECTURE.md).

**Two senses of "bootstrap" — do not conflate them.** This runbook covers both, in
order, but they are different jobs with different operators and lifetimes:

1. **Account bootstrap (the trust anchor)** — **Phases 0–1**. *One-time per AWS
   account*, run by an **admin from a laptop** (default profile, NOT the apply role —
   the apply role doesn't exist yet). Creates the Terraform **state backend** and the
   **identity** roles. After it, the account is ready to be operated by scoped roles.
2. **Platform bring-up (the environment + cluster)** — **Phases 2–5**. Provisions an
   environment's infra, the k3s cluster, and the GitOps apps. Runs **from the
   conductor** under the scoped apply role, driven by `gitops/tools/platform.sh` (whose
   `bootstrap` subcommand brings up the cluster). Repeatable per environment
   (`ENV=dev|prod`). The Ansible step here also "bootstraps" k3s — a third, narrower
   use of the word; it's part of this platform bring-up, not the account bootstrap.

> **Billing:** phases marked **💸** create costed AWS resources. Print the plan
> and confirm before every `apply` (see [../CLAUDE.md](../CLAUDE.md) guardrails).
> Run long phases from the **toolbox container / bootstrap VM** so an expiring
> session can't orphan a half-finished apply.

Status legend: ✅ live · 🔜 pending (built in a later PLAN.md block).

> **Orchestrated path.** The phases below map to `gitops/tools/platform.sh` — one
> gated driver (`ENV=dev|prod`, AWS only) that sequences them. From zero: as **admin
> on the laptop** do Phases 0–1 (state backend + identity), then `platform.sh
> conductor` to launch the ops box and `ship_repo.sh | bash` to push your approved
> working tree to the state bucket (the conductor holds **no GitHub credential** — it
> runs exactly what you ship). Then **SSM onto the conductor**, run `brzl-fetch`
> (pulls the tree via the instance role), and `platform.sh bootstrap` (= `secrets →
> layers(+15-kms) → images → cluster → gitops → watch → roundtrip`). When the
> foundation already exists, `platform.sh all` skips straight to the bring-up. AWS ops
> run **from the conductor** (audited, IAM-gated, identical toolchain); the laptop only
> bootstraps the trust anchor, launches the conductor, and ships the tree. The same
> driver does the DR `restore` and the `teardown`. (local-dev is separate —
> `gitops/clusters/local/k3d_up.sh`, no AWS.)

## Prerequisites
- OpenTofu `>= 1.6` (`tofu`), AWS CLI v2, an admin identity for phases 0–1.
- No SSH key to supply: the node keypair is **generated by OpenTofu**
  (`50-compute/secrets.tf`) and is break-glass only — SSM is the access path.
- The state bucket is **derived, never an env var** (SPEC §3, model B): the stack
  layers carry an empty `backend "s3" {}` and the driver (`platform.sh`) **composes**
  the backend at init — bucket `brzl-demo-tfstate-<account>` from the caller, key
  `<env>/<layer>/terraform.tfstate`. No per-env `backend.hcl`, no `TF_VAR_state_bucket`.
  The cross-layer `terraform_remote_state` data sources derive the same. (The
  account-primitives `terraform/bootstrap`, `terraform/identity`, and the conductor
  keep their own `backend.hcl` — they're not stack layers.)
- The admin /32 lock for SSH/the kube-API is **auto-detected** — the 20-security
  layer fetches the applying host's public IP at plan time (an `http` data source),
  so it works from the conductor or a laptop with no shell export. Pin a fixed range
  only if you need to: `-var admin_cidr=203.0.113.0/24`.

## Phase 0 — State backend · account bootstrap ✅ 💸(~$1/mo)
Creates the S3 state bucket (versioned, **customer-managed KMS** encrypted,
private) + DynamoDB lock table. The CMK is ~$1/mo; state storage itself is ~$0.
```sh
cd terraform/bootstrap
cp terraform.tfvars.example terraform.tfvars   # set globally-unique bucket name
tofu init && tofu plan && tofu apply
```
The bucket name follows `brzl-demo-tfstate-<account_id>` — the driver derives it at
runtime, so the **stack layers need no `backend.hcl`**. (Only the account-primitives
`terraform/identity` + the conductor still take a `backend.hcl`, copied from
`backend.hcl.example`.)

> This layer's own state is deliberately **local** (`terraform/bootstrap/
> terraform.tfstate`, gitignored) — it manages the bucket, so it can't live in
> it. That local-state-then-migrate shape is the model for a generalized
> bootstrap-from-zero (and, reversed, for clean retirement — TEARDOWN.md §3).

## Phase 1 — Identity / trust anchor · account bootstrap ✅ (free)
GitHub Actions OIDC provider + least-privilege `brzl-tofu-plan` / `brzl-tofu-apply`
roles. After this, **subsequent phases assume the apply role** instead of admin.
```sh
cd ../identity
cp backend.hcl.example backend.hcl              # set real bucket (gitignored)
cp terraform.tfvars.example terraform.tfvars    # set operator ARN (gitignored)
tofu init -backend-config=backend.hcl && tofu plan && tofu apply
```
Manual prerequisite (not Terraformed): enable **IAM Identity Center** in the
AWS console/Organizations if you want human SSO. Role/SSO onboarding and the
assume-role workflow are in [ACCESS.md](ACCESS.md).

## Phase 2 — Infrastructure layers · platform bring-up ✅ 💸(NAT gw + 3× m6g + EBS + KMS)
Apply the environment's layers **in order**. The normal path is the driver from
the conductor (gated, saved-plan per layer):
```sh
ENV=dev bash gitops/tools/platform.sh layers     # or ENV=prod
```
The manual per-layer reference (laptop: `AWS_PROFILE=brzl-apply`; conductor:
instance-role creds, no profile) — the **saved-plan workflow, never a blind
loop** (CLAUDE.md). One **single-source** dir per layer; the env is the `-var-file`
+ the composed backend key (model B, SPEC §3):
```sh
acct="$(aws sts get-caller-identity --query Account --output text)"
cd terraform/stack/aws/10-network               # then 15-kms → 20 → 30 → 40 → 50
tofu init -reconfigure \
  -backend-config="bucket=brzl-demo-tfstate-${acct}" \
  -backend-config="key=dev/10-network/terraform.tfstate" \
  -backend-config="region=eu-central-1" \
  -backend-config="dynamodb_table=brzl-demo-tflock" -backend-config="encrypt=true"
tofu plan -var-file=dev.tfvars -out=tfplan      # ← review
tofu apply tfplan                               # (saved plan embeds the vars)
```
`15-kms` is the persistent foundation (CMKs + the CNPG backup bucket); `30-iam`
reads it for the node role's backup grants and `50-compute` reads it for the EBS
CMK. The node SSH keypair is TF-generated in `50-compute` — nothing to supply.
**Checkpoint:** `tofu output` in `50-compute` lists 3 nodes → infra live.
(`ENV=prod`: same driver/dirs, `-var-file=prod.tfvars` + key `prod/<layer>/…` —
private-subnet nodes, and `50-compute` creates the demo-app **NLB**
(`enable_public_ingress=true`). The driver does all this for you.)

## Phase 3 — Node config & k3s ✅ (validated live 2026-06-04 / 06-09)
Render the Ansible inventory from `50-compute` outputs, then
`ansible-playbook playbooks/bootstrap.yml` (base → security) and
`playbooks/cluster.yml` (k3s HA). Driver: `platform.sh cluster`.
**Checkpoint:** `kubectl get nodes` = 3 Ready. (`ENV=prod`: nodes are private —
kubectl reaches the API through an **SSM port-forward** to a node's 6443,
kubeconfig server `https://127.0.0.1:6443`; 127.0.0.1 is in k3s's default TLS
SANs. See [ACCESS.md](ACCESS.md).)

## Phase 4 — GitOps ✅ (validated live 2026-06-04 / 06-09)
**Prereqs (one-time):**
- **Registry pull-through creds** — images come from ghcr.io (CNPG, external-
  secrets) and Docker Hub (Grafana), which require **authenticated** ECR pull-through
  (Secrets Manager — ECR can't use Parameter Store for this). quay.io and
  registry.k8s.io are credential-free. Create the secrets with the helper for
  whichever upstreams you need, then pass the printed ARNs to `40-ecr` and re-apply
  (💸 negligible). The creds are added incrementally as each layer needs them — ghcr
  at the CNPG block, Docker Hub at the monitoring block:
  ```sh
  export GHCR_USERNAME=... GHCR_TOKEN=...              # GitHub user + read:packages PAT
  export DOCKERHUB_USERNAME=... DOCKERHUB_TOKEN=...    # Docker Hub user + read-only token
  AWS_PROFILE=brzl-apply bash gitops/bootstrap/create_pullthrough_secrets.sh        # preview
  AWS_PROFILE=brzl-apply bash gitops/bootstrap/create_pullthrough_secrets.sh | bash # run
  # render the resulting ARNs straight into the 40-ecr tfvars (no paste-back), then apply:
  AWS_PROFILE=brzl-apply bash gitops/bootstrap/create_pullthrough_secrets.sh \
    write_arns_to_tfvars terraform/stack/aws/40-ecr/credentials.auto.tfvars   # account-level; shared dev/prod
  ```

  **Minting the tokens — exact shapes.** Each credential does one job; none can
  do another's (deliberate — see [SECURITY.md](SECURITY.md)):
  - **`GHCR_TOKEN`** (registry pulls): GitHub → Settings → Developer settings →
    Personal access tokens → **Tokens (classic)** → scope **`read:packages` only**,
    short expiry. No repo scopes — this token deliberately *cannot* clone the repo
    (the A2 run confirmed a clone attempt 403s). The fine-grained UI is the default;
    note that repo permissions there (Contents/Metadata) do **not** grant registry
    pulls — the classic `read:packages` scope is the validated shape for ECR
    pull-through.
  - **`DOCKERHUB_TOKEN`**: hub.docker.com → Account Settings → Personal access
    tokens → generate with **Read-only** access.
  - **Repo *read* credential** (anything that must clone the private repo over
    HTTPS — not currently needed: ArgoCD uses the SSH deploy key below, and the
    conductor gets the tree via S3): a **fine-grained PAT**, Only select
    repositories → this repo, Repository permissions **Contents: Read-only** (+
    Metadata, auto-included), nothing else.
  40-ecr also publishes the registry host to SSM Parameter Store
  (`/brzl-dev/ecr/registry_host`) — the bootstrap reads it from there.
- **Repo deploy key** *(must pre-stage — see below)* — `ssh-keygen -t ed25519` a
  read-only key, add the public half to the GitHub repo (Settings → Deploy keys,
  **no** write), copy `gitops/bootstrap/repo-deploy-key.example.yaml` →
  `repo-deploy-key.yaml` (real one is gitignored), paste the private half. Without
  it ArgoCD installs but can't sync the private repo.
  > **Pre-staging on the conductor.** This file is **gitignored**, so it is not in a
  > fresh clone and `ship_repo.sh` does **not** carry it to the conductor — place it on
  > whichever host runs the `gitops` phase (the conductor), e.g. from the laptop:
  > `scp -i ~/.ssh/<conductor-key> -o IdentitiesOnly=yes -o ProxyCommand='aws ssm
  > start-session --target <conductor-id> --document-name AWS-StartSSHSession
  > --parameters portNumber=%p' gitops/bootstrap/repo-deploy-key.yaml
  > ec2-user@<conductor-id>:/opt/brzl/brzl-demo/gitops/bootstrap/`. The keypair is
  > generated **once**; the public half stays registered on GitHub across conductor
  > rebuilds, so only the private-half file needs re-placing. (Evaluators pointing
  > ArgoCD at a fork: also update the `url:` in the Secret and the `ApplicationSet`'s repo.)
  >
  > **The symptom when it's missing** (it has bitten two live runs): the `gitops`
  > phase succeeds, ArgoCD runs — and **every Application shows `SYNC: Unknown` with
  > an empty `REVISION`**, because ArgoCD has no credential to read the private repo.
  > The driver now **blocks** on this before bootstrapping. To fix after the fact:
  > stage the file and `kubectl apply -f gitops/bootstrap/repo-deploy-key.yaml` —
  > the repo-server picks it up within ~3 minutes, no restart needed.
- **Grafana admin secret** — the monitoring app's Grafana reads its admin login
  from an in-cluster Secret (`admin.existingSecret`), so the password never lands
  in git and stays stable (the chart's own default regenerates each render). **The
  `gitops` phase of `platform.sh` creates this for you** (before the monitoring wave
  syncs, so Grafana never stalls on a missing Secret) — `export GRAFANA_ADMIN_PASSWORD`
  beforehand to pin a known password, otherwise one is generated. To (re)create it
  out of band, the helper is standalone (emit-commands; it makes the namespace too):
  ```sh
  bash gitops/bootstrap/create_cluster_secrets.sh        # preview
  bash gitops/bootstrap/create_cluster_secrets.sh | bash # run (generates the password)
  ```
  `gitops/tools/ui_forward.sh` reads it back to print the login.

**Install** (emit-commands script; review then pipe to a shell):
```sh
AWS_PROFILE=brzl-apply bash gitops/bootstrap/bootstrap_argocd.sh        # preview
AWS_PROFILE=brzl-apply bash gitops/bootstrap/bootstrap_argocd.sh | bash # run
```
It helm-installs ArgoCD (image via ECR pull-through), waits for rollout, applies
the deploy-key Secret + the `brzl-dev` AppProject + the **in-cluster cluster
Secret** (annotating it with the ECR host and backup bucket) + the **`ApplicationSet`**,
and makes gp3 the single default StorageClass (drops k3s local-path's default
annotation). ArgoCD then self-manages and syncs by wave: argocd (-1) → EBS CSI +
gp3 (0) → operators (cnpg, external-secrets, 1) → Postgres (2) → demo-app (3).

**Account-id note (ADR-0016):** no account id is committed. The host is derived
live from `aws sts get-caller-identity` (`resolve_ecr_host.sh`): injected into the
ArgoCD install via `helm --set`, and written (with the backup bucket) onto the
in-cluster Secret's annotations. The **ApplicationSet's `clusters` generator** reads
those annotations and injects the real host/bucket at render — Helm `parameters`
for the operator charts, kustomize `images`/`patches` for the workloads. The
committed `__ECR_REGISTRY_HOST__` / `__BACKUP_BUCKET__` sentinels are documentation
only.

Reach the UI without a cloud LB: `kubectl -n argocd port-forward svc/argocd-server
8080:80` (admin password: `kubectl -n argocd get secret argocd-initial-admin-secret
-o jsonpath='{.data.password}' | base64 -d`).

**Checkpoint:** ArgoCD shows the infrastructure apps Synced/Healthy; `kubectl get
sc` shows `gp3 (default)`.

## Phase 5 — Operator + DB + app ✅ (validated live 2026-06-04 / 06-09)
CloudNativePG operator → `Cluster` CR (3 instances, gp3 PVCs) → demo-app, all
via GitOps. **Checkpoint:** demo-app reads/writes Postgres end to end —
`seed_demo_data.sh pf` drives a full search→DB roundtrip. (`ENV=prod`: the UI
is reached via the **NLB** — `tofu output demo_app_url` in prod `50-compute`;
the allowlist is the operator /32 unless `lb_ingress_cidr` was widened.)

## If something breaks
- Apply fails mid-layer → fix, re-run that layer's `apply` (idempotent); state lock
  stuck? see [RECOVERY.md](RECOVERY.md#state-lock-stuck).
- Never hand-edit cloud resources Terraform manages — reconcile via code.
